# Technical Writer: 技術ドキュメント執筆支援

## Overview

技術記事・ドキュメントを対話形式で作成。企画→構成→執筆→推敲→ファイル生成まで一貫してサポート。

**Core principle:** 読者にとって価値のある、構造化された明確なドキュメントを効率的に作成する。

## When to Use

- 技術ブログ記事の執筆（Zenn, Qiita, note）
- プロジェクトREADMEの作成・更新
- API仕様書・設計ドキュメントの作成
- チュートリアル・ガイドの執筆
- 社内ドキュメント・ナレッジベースの整備

## The Process

### Phase 1: 企画・要件定義

**目的とターゲットの明確化:**

1. **誰に向けて書くか？**
   - 初心者/中級者/上級者
   - 技術スタック・前提知識
   - 読者が抱える課題

2. **何を伝えたいか？**
   - 主題（1つに絞る）
   - 読者が得られる価値
   - 読後のアクション

3. **どこで公開するか？**
   - Zenn（技術者コミュニティ）
   - Qiita（技術情報共有）
   - GitHub README（プロジェクト説明）
   - 社内Wiki（ナレッジ共有）

```markdown
## 企画テンプレート

**タイトル案:** [具体的で検索されやすいタイトル]
**ターゲット:** [想定読者]
**前提知識:** [必要な知識]
**読者の課題:** [解決したい問題]
**提供価値:** [読むことで得られるもの]
**公開先:** [Zenn/Qiita/README/etc]
```

### Phase 2: 構成案作成

**記事構成の基本パターン:**

```markdown
# タイトル

## リード文（100-200字）
- 記事の概要
- 対象読者
- 読むメリット

## 本文

### 1. 導入・背景
- なぜこのトピックが重要か
- 課題の明確化

### 2. 解決策・本題
- 具体的な方法・手順
- コード例・図解

### 3. 実践・応用
- ハンズオン
- ユースケース

### 4. まとめ
- 要点の整理
- 次のステップ

## 参考資料
```

**構成作成のポイント:**

- 各セクション200-400字を目安
- H2は3-5個、H3は各H2に2-4個
- コード例は動作確認済みのもの
- 図解・スクリーンショットの挿入ポイントを決める

### Phase 3: 執筆

**表記ルール:**

| ルール | 例 |
|--------|-----|
| 文体統一 | です・ます調 or である調 |
| 英単語前後スペース | 正: `React を使用` 誤: `Reactを使用` |
| コード記法 | インラインは`backtick`、ブロックは言語指定 |
| 見出し | 動詞で始める or 名詞で統一 |
| 箇条書き | 体言止め or 動詞で統一 |

**執筆フロー:**

1. **ドラフト作成**
   - 構成に沿って一気に書く
   - 完璧を求めない
   - コード例は後から確認

2. **セクションごと確認**
   - 200-300字ごとに区切って提示
   - ユーザーにレビュー依頼
   - フィードバック反映

3. **コード例の検証**
   - 実際に動作確認
   - エラー処理の追加
   - コメント追加

### Phase 4: 推敲

**チェックリスト:**

```markdown
## 内容
- [ ] タイトルと内容が一致している
- [ ] 主題が1つに絞られている
- [ ] 読者の課題が解決される
- [ ] 具体例・コードが動作する

## 構成
- [ ] 論理的な流れになっている
- [ ] 各セクションの長さが適切
- [ ] 不要な重複がない

## 表現
- [ ] 文体が統一されている
- [ ] 専門用語に説明がある
- [ ] 冗長な表現を削除

## 技術的正確性
- [ ] コードが最新のベストプラクティス
- [ ] セキュリティ上の問題がない
- [ ] パフォーマンスの考慮
```

### Phase 5: ファイル生成

**Zenn形式:**

```markdown
---
title: "タイトル"
emoji: "📝"
type: "tech" # tech or idea
topics: ["topic1", "topic2"]
published: false
---

本文...
```

**Qiita形式:**

```markdown
---
title: タイトル
tags:
  - tag1
  - tag2
private: false
---

本文...
```

**README形式:**

```markdown
# プロジェクト名

概要（1-2文）

## 特徴

- 特徴1
- 特徴2

## インストール

\`\`\`bash
npm install package-name
\`\`\`

## 使い方

\`\`\`typescript
// 基本的な使用例
\`\`\`

## ライセンス

MIT
```

## 対話フロー

```
1. ユーザー: 「〇〇について記事を書きたい」

2. Claude: 企画テンプレートを提示し、質問
   - 「ターゲット読者は誰ですか？」
   - 「読者が抱える課題は何ですか？」

3. ユーザー: 回答

4. Claude: 構成案を提示
   - 「この構成でいかがでしょうか？」

5. ユーザー: フィードバック

6. Claude: 各セクションを順に執筆
   - 200-300字ごとに確認

7. ユーザー: レビュー・修正依頼

8. Claude: 推敲チェックリスト実行

9. Claude: 最終ファイル生成
   - 指定フォーマットで出力
```

## Key Principles

- **One topic per article** - 主題を1つに絞る
- **Reader-first** - 読者の課題解決を最優先
- **Concrete examples** - 抽象論より具体例
- **Iterative validation** - セクションごとに確認
- **Working code** - コード例は動作確認済み

## Output Locations

| 公開先 | 出力パス |
|--------|----------|
| Zenn | `articles/YYYY-MM-DD-slug.md` |
| Qiita | `qiita/YYYY-MM-DD-title.md` |
| README | プロジェクトルート `README.md` |
| Docs | `docs/` 配下 |

## Integration

- **brainstorming** skill: 記事のアイデア出しに活用
- **code-reviewer** skill: コード例のレビューに活用
