# Task Planner: タスク細分化・実装計画策定

## Overview

曖昧な要望や大きな機能要件を、ファイル単位の具体的なToDoリストに分解。タスク間の依存関係を整理し、実行可能な実装計画を策定。

**Core principle:** 曖昧さを排除し、実行可能な粒度まで分解することで、確実な実装を実現する。

## When to Use

- 新機能の実装計画
- 大規模リファクタリングの計画
- バグ修正の調査・計画
- 技術的負債の解消計画
- プロジェクト見積もり

## Planning Process

### Phase 1: 要件の明確化

**質問テンプレート:**

```markdown
## 要件確認

1. **ゴール**
   - 何を実現したいですか？
   - 成功の定義は何ですか？

2. **スコープ**
   - 含まれる機能は？
   - 含まれない機能は？
   - 既存機能への影響は？

3. **制約**
   - 技術的制約は？
   - 時間的制約は？
   - 既存コードとの互換性は？

4. **優先度**
   - Must Have / Should Have / Nice to Have
   - 最小限のMVPは何か？
```

### Phase 2: 既存コード調査

**調査項目:**

```markdown
## コードベース調査

### 関連ファイル特定
- [ ] 変更対象ファイルの特定
- [ ] 依存関係の把握
- [ ] 影響を受けるテストの特定

### パターン理解
- [ ] 既存の実装パターン確認
- [ ] 命名規則の確認
- [ ] ディレクトリ構造の理解

### 技術スタック確認
- [ ] 使用ライブラリの確認
- [ ] バージョン互換性の確認
- [ ] 設定ファイルの確認
```

**調査コマンド:**

```bash
# 関連ファイル検索
rg "keyword" --type ts

# ディレクトリ構造確認
eza --tree --level=3 src/

# 依存関係確認
npm ls <package-name>

# 型定義確認
rg "interface|type" --type ts src/types/
```

### Phase 3: タスク分解

**分解原則:**

```markdown
## 分解の原則

### 粒度
- 1タスク = 1ファイル または 1機能単位
- 完了判定が明確
- 独立してテスト可能

### 依存関係
- 前提タスクを明確化
- 並列実行可能なタスクを特定
- クリティカルパスを把握

### 見積もり基準
- S: 30分以内
- M: 1-2時間
- L: 半日
- XL: 1日以上（要分割）
```

**タスク分解テンプレート:**

```markdown
## タスク分解

### カテゴリ1: [機能/レイヤー名]

#### Task 1.1: [タスク名]
- **ファイル**: `src/path/to/file.ts`
- **概要**: 何をするか
- **依存**: なし / Task X.X
- **サイズ**: S/M/L
- **詳細**:
  - [ ] サブタスク1
  - [ ] サブタスク2

#### Task 1.2: [タスク名]
...
```

### Phase 4: 依存関係マッピング

**依存関係図:**

```markdown
## 依存関係

\`\`\`mermaid
graph TD
    A[Task 1: 型定義] --> B[Task 2: Repository]
    A --> C[Task 3: Service]
    B --> D[Task 4: API Handler]
    C --> D
    D --> E[Task 5: テスト]
    D --> F[Task 6: ドキュメント]
\`\`\`

### クリティカルパス
A → B → D → E

### 並列実行可能
- Task 2, Task 3（Task 1完了後）
- Task 5, Task 6（Task 4完了後）
```

```markdown
- [ ] ドキュメント
- [ ] コードレビュー
```

```markdown
## バグ修正: [Issue番号]

### Phase 2: 修正
- [ ] 修正実装

- [ ] ローカル検証
```

### リファクタリング

```markdown
## リファクタリング: [対象]

### Phase 1: 現状分析
- [ ] 問題点の特定
- [ ] 依存関係の把握
- [ ] テストカバレッジ確認

### Phase 2: 段階的改善
- [ ] ステップ1: [小さな改善]
- [ ] ステップ2: [次の改善]
- [ ] ステップ3: [最終形]
```

## TaskCreate/TaskUpdate Integration

**タスク登録:**

TaskCreateツールで各タスクを登録し、TaskUpdateで依存関係（addBlockedBy）を設定。
各タスクにはsubject、description、activeFormを指定する。

## Key Principles

- **Concrete over abstract** - 抽象的な計画より具体的なタスク
- **File-level granularity** - ファイル単位で分解
- **Clear dependencies** - 依存関係を明確に
- **Testable tasks** - 各タスクが検証可能
- **Incremental delivery** - 段階的にデリバリー

## Integration

- **brainstorming** skill: 要件の明確化に活用
- **ddd** skill: ドメイン駆動での分解
- **kaizen** skill: 段階的改善の視点
- **subagent-drive-development** skill: タスク並列実行
